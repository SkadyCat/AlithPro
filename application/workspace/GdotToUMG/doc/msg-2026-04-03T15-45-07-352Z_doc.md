## 功能映射结论

| Godot 侧 | UMG 侧 | 功能映射 | 转换注意点 |
| --- | --- | --- | --- |
| `Control` | `Widget` | 两者都是 UI 基类，负责可见性、焦点、输入、变换、裁剪等通用能力。 | Godot 的锚点/偏移是所有 `Control` 的基础能力；UMG 的绝对布局通常要落到 `CanvasPanelSlot`。 |
| `Container` | `PanelWidget` | 两者都负责容纳并布局子控件。 | Godot 由父 `Container` 主导子项布局；UMG 常由具体 `Slot` 类型保存布局参数。 |
| `Control + anchors/offsets` | `CanvasPanel + CanvasPanelSlot` | 都适合绝对/锚点式 HUD 布局。 | 这是 Godot 锚点布局最接近的 UMG 落点；不适合大规模自动流式布局。 |
| `HBoxContainer` / `VBoxContainer` | `HorizontalBox` / `VerticalBox` | 都是线性布局容器。 | Godot 通过 `Expand`、`Fill`、`Stretch Ratio` 分配空间；UMG 需要映射到对应 slot 的 size / fill 规则。 |
| `GridContainer` | `GridPanel` / `UniformGridPanel` | 都是网格布局。 | 如果 Godot 侧每格尺寸统一，更适合 `UniformGridPanel`；否则用 `GridPanel` 保留行列控制。 |
| `MarginContainer` | `Border` / 带 padding 的容器 | 都用于给内容增加内边距。 | Godot 的 margin 经常来自 Theme 常量；UMG 要改写为 padding/slot padding。 |
| `PanelContainer` / `Panel` | `Border` | 都能提供背景绘制并包裹内容。 | Godot 偏 `StyleBox`；UMG 偏 `Brush` + padding。视觉资源需要单独换算。 |
| `ScrollContainer` | `ScrollBox` | 都为超出区域的内容提供滚动。 | 两者都适合中小规模内容；UMG `ScrollBox` 文档明确说明不支持 virtualization，大列表应转 `ListView`。 |
| `TabContainer` | `WidgetSwitcher + tab button bar` | 功能上都表现为“多页签 + 当前页可见”。 | UMG 没有与 Godot `TabContainer` 完全等价的一体化控件，通常要把页签头和内容切换拆开实现。 |
| `Label` | `TextBlock` | 都负责静态文本显示。 | 对齐、自动换行、字体样式需要做主题字段映射。 |
| `RichTextLabel` | `RichTextBlock` | 都支持富文本。 | Godot 偏 BBCode/tag stack；UMG 偏 Rich Text Decorator 体系，标记语法和扩展点不同。 |
| `TextureRect` | `Image` | 都用于显示纹理/图片。 | Godot `stretch_mode` 需要换算到 UMG brush draw / fit / padding 组合；覆盖式裁剪通常还要配合容器。 |
| `Button` | `Button` | 都是基础可点击控件。 | Godot 常把图标/文本作为子控件叠上去；UMG 文档也说明 `Button` 可包一个子 widget，结构上较容易对齐。 |
| `LineEdit` | `EditableTextBox` | 都是单行输入框。 | 焦点、提交、占位文本、只读/可编辑状态需要逐项映射。 |
| `TextEdit` | `MultiLineEditableTextBox` | 都是多行编辑器。 | Godot `TextEdit` 更接近编辑器式文本控件，功能常比 UMG 默认多；高级编辑行为可能只能部分保留。 |
| `CheckBox` | `CheckBox` | 都表达二值切换。 | Godot 常带文字；UMG 往往要额外组合 `TextBlock`。 |
| `OptionButton` / `MenuButton` | `ComboBoxString` / `MenuAnchor` | 都可表达下拉选择或菜单入口。 | Godot 的菜单/弹出结构与 UMG 数据绑定方式不同，通常需要转换为“数据源 + 选项模板”。 |
| `ProgressBar` / `TextureProgressBar` | `ProgressBar` | 都表示进度。 | `TextureProgressBar` 的分层贴图与径向样式不能直接平移到 UMG 默认进度条，常需自定义材质或样式。 |
| `HSlider` / `VSlider` | `Slider` | 都表示连续值滑条。 | 方向、步长、样式和手柄贴图需要单独映射。 |
| `ItemList` | `ListView` / `TileView` / `ScrollBox + item widget` | 都可表达可滚动列表。 | 如果后续目标是大数据量，优先转 `ListView/TileView`，不要直接降级成 `ScrollBox`。 |
| `Tree` | `TreeView` | 都表达层级数据。 | Godot `Tree` 是现成控件；UMG 常更偏数据驱动，节点模板与展开状态管理要额外实现。 |

## 关键差异与转换规则

### 1. 布局模型差异最大

- Godot 的 `Control` 天生带锚点、偏移、最小尺寸与容器协作能力。
- UMG 更明显地区分“控件本体”和“父容器 slot 布局参数”。
- 因此转换时不能只做控件名替换，必须同时生成 **父容器类型 + slot 参数**。

### 2. 容器选择比控件名称更重要

- Godot 场景里很多视觉结果其实是由 `Container` 规则驱动的。
- 如果仅把 `Label`、`Button`、`TextureRect` 映射到 UMG 同名控件，但没有正确补齐 `CanvasPanel`、`HorizontalBox`、`VerticalBox`、`GridPanel` 等父层结构，最终布局会偏差很大。

### 3. 主题系统不能一比一照搬

- Godot 常用 `Theme`、`StyleBox`、主题常量、字体覆写。
- UMG 常用 `FSlateBrush`、`FButtonStyle`、`FTextBlockStyle`、padding 与状态样式。
- 所以样式转换应拆成：
  1. **结构映射**：先决定目标 widget 树。
  2. **布局映射**：再落锚点、padding、fill、alignment。
  3. **视觉映射**：最后落贴图、字体、颜色、状态刷子。

### 4. 复杂控件建议按“能力近似”而非“名字相同”处理

- `TabContainer`、`ItemList`、`Tree`、`TextureProgressBar` 这类控件在两边的现成能力并不完全对齐。
- 这类节点更适合按“页签切换”“数据列表”“层级视图”“贴图化进度条”四类能力做二次组装，而不是机械替换成单一 UMG 控件。

## 对 GdotToUMG 的直接建议

1. 先建立 **基类映射表**：`Control -> Widget`、`Container -> PanelWidget`。
2. 再建立 **布局优先级规则**：
   - 有 anchors/offsets 的节点优先落 `CanvasPanelSlot`
   - 处于 `HBoxContainer/VBoxContainer/GridContainer` 子树时优先保留线性/网格容器结构
3. 对高风险控件单独分流：
   - `TabContainer`
   - `ItemList`
   - `Tree`
   - `TextureProgressBar`
4. 主题不要首轮强求完全还原，先保证结构、交互、尺寸关系正确，再逐步补视觉。
