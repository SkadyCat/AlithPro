调研结论：
1. **社区/平台的主流逻辑不是“一次生成最终 UI”**，而是把流程拆成：
   - 基础模型 / 自定义模型（或 LoRA）
   - 提示词模板与负面提示词
   - 参考图 / 结构控制
   - 节点化工作流
   - 批量生成与后处理
2. **风格一致性比单张效果更重要**。Scenario 强调用 10-50 张参考图训练自定义模型或风格模型，核心价值是让整套资源保持统一视觉语言，而不是每次重新碰运气。
3. **节点化工作流已经成为生产标配**。Scenario Workflows 和 ComfyUI 都强调“模型节点 + 控制节点 + 后处理节点”的可组合流程，适合把草图、参考图、修图、放大、抠图、批处理接成稳定流水线。
4. **批量变体生成非常关键**。对游戏 UI 来说，同一控件经常要产出 common / rare / epic / legendary、不同主题、不同尺寸和不同状态（idle / hover / active / disabled）版本，平台普遍支持一套设定批量出多张，再统一筛选。
5. **AIGC 更适合产出“风格层”和“装饰层”**，不适合直接替代所有功能性 UI。像按钮边框、面板纹样、徽章、浮雕、背景底纹、图标概念稿，很适合用 AIGC；但九宫格切片、对齐精度、文字可读性、交互态规范，仍然需要规则化加工。

对当前仓库的建议：
1. **继续保留现有确定性生成链**  
   当前 `application\godot_inventory\tools\generate_oriental_inventory_assets.js` 与 `generate_oriental_chrome_assets.js` 的价值是稳定、可复现、可批量，不应被完全替换。
2. **把 AIGC 放到“视觉母版”层**  
   更合理的做法是先用 AIGC 生成：
   - 面板花纹母版
   - 按钮金属/玉石/科幻包边母版
   - 徽章、印章、纹章、云纹、能量纹理
   - 图标风格探索稿  
   再由现有 Node + Canvas 脚本把这些母版裁切、组合、标准化输出为最终控件资源。
3. **以 `bag.bp` 为资源需求清单**  
   `bag.bp` 已经列出面板、标签、按钮、装备槽、详情底图、图标等资源。下一步可以把它扩成“资源规格表”，为每个资源增加：
   - 主题（国风 / 科幻 / 轻快）
   - 状态（idle / active / disabled）
   - 生成方式（AIGC 母版 / 脚本绘制 / 手工修图）
   - 是否需要九宫格
4. **建议采用“三段式流水线”**
   - **阶段 A：AIGC 探索**  
     用 LoRA / 参考图 / 负面提示词，先出 20-50 张风格探索图，目标是找到统一语言，而不是直接上线。
   - **阶段 B：结构约束**  
     用参考图、草图、遮罩或 ComfyUI / Scenario 工作流，把面板、按钮、边框、印章控制在稳定结构里。
   - **阶段 C：工程化落地**  
     进入脚本裁切、透明背景、尺寸归一、九宫格、安全边距和 Godot 导入阶段。
5. **建议优先 AIGC 化的资源类型**
   - `topbar_crest.png`
   - `title_plate.png`
   - `currency_plate.png`
   - `grid_header_plate.png`
   - `detail_seal.png`
   - `detail_hero.png`
   - `detail_title_plate.png`
   - `detail_lore_plate.png`
   - `equipment_slot*.png`
   - 各类 `item_*` 图标  
   这些资源视觉风格权重大，且比纯功能容器更适合通过 AIGC 提升质感。

建议的提示词与流程策略：
- **提示词结构**：材质 + 题材 + 控件类型 + 装饰语言 + 光影 + 背景约束  
  例如：`ornate chinese fantasy game UI panel, carved bronze frame, jade inlay, warm parchment center, symmetrical decorative corners, transparent background, high detail`
- **负面提示词**：避免人物、场景透视、文字、错位、非对称、脏污和过强体积光。
- **参考图策略**：优先喂给模型当前仓库已有的 `inventory_panel_bg.png`、`detail_panel_bg.png`、`topbar_bg.png` 等，保持结构不跑偏。
- **批处理策略**：一次生成同一控件的 8-15 个变体，只保留最稳定的边框、材质和纹样方案。
- **后处理策略**：生成后统一做抠图、透明底、边缘清理、对称修正、尺寸归一，再进入 Godot。

针对当前问题的直接判断：
- 现在“丑”的核心不在于缺资源，而在于**现有资源多为脚本直绘基础版，缺少高品质纹样、材质层次和主题细节**。
- 最优解不是丢掉现有资源脚本，而是把 **AIGC 用于提升视觉母版**，再让现有脚本负责工程化输出。
