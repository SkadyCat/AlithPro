# 交付说明 — 直射 + 环形分裂无效果修复

## 源任务文件
`msg-2026-04-06T12-24-04-808Z.md`

## 目标
修复"直射 + 环形分裂没有效果"的问题。

## 根因分析
Lua 被动类（MagicStraitBall、MagicProjectileRing 等）在从 AngelScript 迁移时只保留了数据属性，**未移植 `process()` 方法**。`skill_comp.lua:268` 检查 `if passive and passive.process then` 对所有被动都为 false，跳过处理。

### 两个被动各自的问题
| 被动 | AS 原版 Process() | Lua 状态（修复前） | Lua 状态（修复后） |
|------|-------------------|-------------------|-------------------|
| **直射 (StraightShot)** | 设置子弹速度和方向 | 无 process() → 子弹速度=(0,0,0) 不移动 | MagicStraitBall:process() 设置 Velocity |
| **环形分裂 (RingSplit)** | 添加 BulletRingBehavior 到子弹 | 无 process() → 无环形行为 | MagicProjectileRing:process() 调用 AddBehavior |

## 改动

### 1. MagicStraitBall.lua（Task 34 已完成）
- 添加 `process(actor, bullet, params)` — 设置 `ProjectileMovement.Velocity`
- 从 `params.Dir` 获取方向，归一化后乘以速度
- 设置 MaxSpeed=3000, InitialSpeed=3000, GravityScale=0
- 设置子弹旋转对齐速度方向

### 2. MagicProjectileRing.lua（本次新增）
- `Content/Script/com/ability/node/.../MagicProjectileRing.lua`
- `Script/com/ability/node/.../MagicProjectileRing.lua`

添加 `process(actor, bullet, params)` 方法：
1. `UE.UClass.Load("/Script/Angelscript.BulletRingBehavior")` 加载 AS 行为类
2. `bullet:AddBehavior(ring_class)` 添加环形行为到子弹
3. 配置行为属性：ProjectileCount、Radius、TriggerDistance、bTriggerOnCollision
4. 属性读取兼容 camelCase/PascalCase（RingSplit 用 camelCase，父类用 PascalCase）

### 环形分裂工作流
1. Lua skill_comp.lua 生成子弹
2. StraightShot.process() → 设置子弹飞行速度/方向
3. RingSplit.process() → 添加 BulletRingBehavior（triggerDistance=1，几乎立即触发）
4. Bullet.as Tick() → BulletRingBehavior.TickBehavior() 检测飞行距离
5. 距离 ≥ triggerDistance → DoRingRelease() 创建 4 个环形投射物
6. 环形投射物通过 AS SkillManager.DoSkillByCustomArray() 生成（继承原技能链，排除环形被动避免递归）

## 验证
- 代码逻辑与 AS 原版 `UMagicProjectileRing.Process()` 一致
- 两目录文件已同步
- AddBehavior 是 ABullet 的 BlueprintCallable UFUNCTION，UnLua 可调用
- Debug print 输出可在 UE Output Log 中验证

## 风险
- `UE.UClass.Load("/Script/Angelscript.BulletRingBehavior")` 路径可能需要调整（如果 AS 类注册路径不同）— 有 debug print 辅助排查
- 环形投射物通过 AS SkillManager 创建，走 AS 执行路径（非 Lua）— 这与原始 AS 流程一致
- CurveHoming（曲线追踪）继承了 MagicStraitBall.process()，但 AS 原版有额外 FindNearestActor 逻辑，后续需要覆盖
