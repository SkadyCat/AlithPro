# 交付说明：UE 崩溃分析 — DoSkillByCustomArray passive missing: 0

## 源任务文件
`msg-2026-04-06T14-03-14-473Z.md`

## 任务目标
分析 UE Editor 崩溃原因：ring release 后 DoSkillByCustomArray 看到所有 passive 的 type=0，导致弹道级联创建，最终在 `FText::Format → GetDebugStringForWorld` 中发生内存访问冲突 (0xC0000005)。

## 根因分析

### 崩溃链
1. Lua `CopySkillInfo(train._val)` 成功复制 skillLinks 数量 (13)，但每个 FPoint2D 元素的 type 字段均为 0
2. Ring release 创建 4 个子弹道，每个子弹道的 OriginalSkillInfo 中被动 type 全为 0
3. DoSkillByCustomArray 对每个子弹道输出 "passive missing: 0" × N
4. Split behavior 可能在子弹道上触发（如果子弹道继承了 split 被动），创建更多孙弹道
5. 弹道级联增长 → 内存压力 → `GetDebugStringForWorld` 的 FText::Format 访问已销毁/无效的 World 数据 → 崩溃

### 根本原因
UnLua 对 TArray<FStruct> 元素的跨语言传递存在浅拷贝缺陷：
- `CopySkillInfo(train._val)` 将 Lua 侧的 FSkillDataInfo userdata 传给 AS UFUNCTION
- AS 端接收到的 struct reference 的顶层字段正确 (skillLinks.Num()=13)
- 但嵌套 TArray 中的 FPoint2D 元素的各字段 (x, y, type) 均为默认值 0
- 这是 UnLua marshaling 的已知限制，nested TArray<FStruct> elements lose field values

## 修改内容

### 1. BaseMasgicCharacter.as — 新增 CopySkillChainToBullet UFUNCTION
- **位置**: 第 446-464 行
- **逻辑**: 直接从角色的 `skill_comp_skill_data.skillDataInfos[InfoIndex]` 逐字段复制到 `Bullet.OriginalSkillInfo`
- **优势**: 完全在 AS 侧完成，绕过 UnLua 的 TArray element marshaling
- 逐个 FPoint2D 创建新实例并复制 x, y, type

### 2. skill_comp.lua — 替换 CopySkillInfo 调用
- **位置**: 第 256 行
- 替换: `bullet:CopySkillInfo(train._val)` → `self.entity.actor:CopySkillChainToBullet(bullet, 0)`
- 添加诊断日志：打印 copied count 和前 5 个 skillLinks 的 type 值

### 3. RingBulletBehavior.as — 添加诊断日志
- **位置**: 第 137-142 行
- DoRingRelease 进入时打印 skillLinks count、ChainBlockedAt、前 5 个 type 值
- 用于验证子弹道接收到的数据是否正确

## 验证方法
需要 UE AS 热编译 (Ctrl+Shift+F5 或重启编辑器) 后测试：
1. 使用包含 Ring 被动的技能
2. 检查 MagicWorld.log 中的诊断日志：
   - `[SKILL_DBG] bullet.skillLinks[N].type = <非零值>` → 修复成功
   - `DoRingRelease: skillLinks[N].type = <非零值>` → 子弹道数据正确
3. 不应再出现 "passive missing: 0" 的级联警告
4. 不应崩溃

## 剩余风险
- **需要 UE 热编译**: 所有 AS 更改 (CopySkillChainToBullet, ChainBlockedAt, 诊断日志) 需重新编译
- **SplitBulletBehavior 仍使用 type-based 过滤**: 当前单 split 场景下可行，后续多 split 场景需改为 index-based
- **诊断日志量较大**: 确认修复后应删减 `[SKILL_DBG]` 日志
- **CurveHoming 等被动缺少 process()**: 后续任务
