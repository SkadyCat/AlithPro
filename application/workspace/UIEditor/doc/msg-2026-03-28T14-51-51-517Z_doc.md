## UI / Lua 逻辑分析

### 1. UI 脚本与资源的组织方式

- 运行时 Lua UI 脚本主要位于：`G:\GameExPro3\MagicWorld\Content\Script\ui\screen\*.lua`
- UI 公共基类与工具位于：
  - `Content\Script\ui\uicommon.lua`
  - `Content\Script\ui\utils\uihelper.lua`
- UI 管理入口位于：
  - `Content\Script\com\utils\ui_manager.lua`
  - `Content\Script\com\utils\G.lua`
- UI 生成说明、面板 JSON、DataAsset 对应关系位于：
  - `Content\Script\ui\jdata\*.json`
  - `Content\Script\ui\IMPORTANT_DA_SETUP.md`
  - `Content\Script\ui\UnlockedSkillsPanel_README.md`

### 2. Lua 是如何接入项目的

- 项目启用了 `UnLua`，并在 `Config\DefaultGame.ini` 中把以下目录加入打包/脚本分发：
  - `Script`
  - `../Plugins/UnLua/Content/Script`
  - `../Plugins/UnLuaExtensions/LuaSocket/Content/Script`
  - `../Plugins/UnLuaExtensions/LuaProtobuf/Content/Script`
- `Content\Script\com\bind\controller.lua` 与 `Content\Script\com\bind\instance.lua` 都是 `UnLua.Class()` 形式，说明核心控制器和实例逻辑直接绑定到了 UE 对象。
- `controller.lua` 中还提供了 `UnLua.HotReload(...)`，用于脚本热重载。

### 3. UI 管理总入口

- 全局对象 `G` 在 `Content\Script\com\utils\G.lua` 中初始化，其中 `G:init_ui_manager()` 会创建 `G.ui`。
- `G.ui` 的实现位于 `Content\Script\com\utils\ui_manager.lua`：
  - 首次访问 `G.ui.SomePanel` 时，会触发 `__getattr`
  - `__getattr` 会调用 `G.controller:custom_ui(panelName, true, da)` 创建对应界面
- 这意味着 UI 是按需懒加载的，而不是在启动时全部一次性实例化。

### 4. Widget 是怎么和 Lua 类绑定的

- `controller.lua` 里有两条常用创建路径：
  - `create_widget(name, attach)`：按 `UIConf.UIConfMap` 直接创建 Widget，然后 `new(name)` 构造 Lua 对象并调用 `init/on_load`
  - `custom_ui(name, attach)`：先通过底层 `create_ui(name, attach)` 拿到 Widget + DataAsset，再通过 `new(data.lua_name)` 构造 Lua 面板对象
- `IMPORTANT_DA_SETUP.md` 说明了这一层的关键依赖：
  - DataAsset (`UUIBaseDA`) 里至少要配置 `UIClass`、`lua_name`、`PanelName`
  - `create_ui` 会从 HUD/UIDataMap 查 DA，再用 `UIClass` 创建 WBP
  - 然后 Lua 侧根据 `lua_name`（例如 `ui.screen.unlocked_skills`）找到对应脚本类
- 所以最终绑定关系是：
  - `DataAsset.PanelName` → UI 管理器访问名
  - `DataAsset.UIClass` → 实际 Widget Blueprint
  - `DataAsset.lua_name` → Lua 脚本类

### 5. Lua UI 基类生命周期

- 所有面板通常继承 `ui.uicommon`，例如：
  - `ui.screen.unlocked_skills`
  - `ui.screen.login_panel`
  - `ui.screen.panel_main_screen`
- `uicommon.lua` 负责：
  - `init(actor, controller, data)`：挂上 Widget、Controller 和配置数据
  - `show()`：首次展示时自动触发 `on_load()`
  - `hide()` / `destroy()`：负责隐藏或移除 Widget
  - `__getattr`：允许 Lua 直接透传访问 Widget 上的命名控件，如 `self.CloseButton`
- 这是一套很典型的“Lua 面板类包一层 UUserWidget”的写法。

### 6. 典型 UI 面板逻辑

#### `ui.screen.login_panel`

- 文件：`Content\Script\ui\screen\login_panel.lua`
- 逻辑较直接：
  - 在 `on_load()` 里绑定 `LoginButton`
  - 读取 `UsernameInput` / `PasswordInput`
  - 校验输入后拼出登录 URL
  - 调用 `self.controller:open_world(url)` 切图/连服
- 这是一个“表单输入 + 按钮事件 + 跳转”的基础 UI 面板。

#### `ui.screen.panel_main_screen`

- 文件：`Content\Script\ui\screen\panel_main_screen.lua`
- 这是更核心的主界面脚本：
  - `on_load()` 初始化右侧菜单、技能槽、头像和小地图 tick
  - `refresh()` 从 `owner().skill_comp` 读取技能槽配置并刷新列表
  - `on_initialize_right_menu_slot()` 里点击菜单按钮后通过 `G.ui[OpenPanel.PanelName]:show()` 打开其他面板
  - `on_level_loaded()` / `tick_minimap()` 负责小地图状态联动
- 说明主屏 UI 不是纯静态展示，而是直接从实体组件数据和地图配置中刷新。

#### `ui.screen.unlocked_skills`

- 文件：`Content\Script\ui\screen\unlocked_skills.lua`
- 这是最完整的 Lua UI 示例之一：
  - `on_load()` 绑定 Close / Filter / Sort 按钮
  - 初始化 TileView，并监听 `BP_OnEntryInitialized`
  - 从玩家实体的 `skill_comp` 读取已解锁技能
  - 监听 `skill_comp.on_unlocked_skills_changed`，在技能变化时自动刷新 UI
  - 若真实数据缺失，会退回示例数据 `load_example_data()`
- 这说明 UI 层和游戏组件层之间是直接联动的，而不是完全由 Blueprint 中转。

### 7. TileView / ItemWidget 的数据流

- `unlocked_skills.lua` 中把技能数据整理成表后，写入 `self.skills_data`，再刷新 TileView。
- 具体条目脚本是 `Content\Script\ui\screen\skill_item_widget.lua`：
  - 实现 `OnListItemObjectSet(list_item_object)`
  - 从 `list_item_object.custom_data.raw_data` 取真正数据
  - 调用 `bind_data(skill_data)` 更新 `SkillName`、`SkillLevel`、`SkillDesc`、`SkillStats`、`SkillImage`
- 公共辅助方法 `uihelper.list()` 也会：
  - 创建 `UStructItem`
  - 把 `custom_data.raw_data` 塞进去
  - 通过 `BP_OnEntryInitialized` 回调初始化列表条目
- 因此项目的列表 UI 数据流大致是：
  - 游戏组件数据 → Lua 面板整理成显示数据 → UObject/Struct 包装 → TileView → Item Lua 脚本渲染

### 8. UI 与游戏实体/组件的关系

- UI 脚本广泛依赖 `G.player`、`G.controller`、`entities_manager` 和各类组件（如 `skill_comp`、`map_comp`）。
- `instance.lua` 中 `OnLevelInstanceLoaded()` 会把关卡事件转发给 `G.ui.PanelMainScreen:on_level_loaded(track_key)`。
- `init_comp.lua`、`skill_comp.lua`、`space_comp.lua` 也会在数据变化时调用：
  - `G.ui.PanelMainScreen:refresh()`
  - `G.ui.PanelMainScreen:set_map_data(...)`
- 这说明 UI 刷新并非单向“UI 自己轮询”，而是和游戏逻辑组件存在主动调用关系。

### 9. 一个重要结论：项目 UI 不是“纯 Lua”，而是“WBP + DataAsset + Lua 绑定”组合

- WBP 决定控件树和命名控件。
- DataAsset 决定 `UIClass` / `lua_name` / `PanelName` 的映射。
- Lua 脚本负责事件绑定、数据读取、刷新和业务行为。
- `Content\Script\ui\jdata\*.json` 和相关 README 则承担“从设计/JSON 到 WBP/DA/Lua 命名”的生成约定。
